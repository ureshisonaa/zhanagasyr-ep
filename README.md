# ZhanaGasyr Education Platform

Закрытая образовательная платформа для учеников карьерного акселератора ZhanaGasyr —
персональный AI-консультант (RAG) по поступлению в зарубежные университеты, управление
документами через Google Drive, календарь дедлайнов и рабочее пространство по каждой заявке
(Application Workspace).

Статус: **Этап 11.1 — Admin: Users & Roles (frontend)** завершён (см. `ZhanaGasyr_Roadmap.md`,
Фаза 11). Вторая и последняя дыра, найденная при структурном аудите всего приложения — backend
был готов с момента первого прохода Этапа 11.1, фронтенда не было вовсе.

## Admin: Users & Roles (Frontend)

## Деплой — конфигурация (Этап 14.2)

Файлы строго по Roadmap: `frontend/vercel.json`, `backend/railway.json`, `backend/Procfile`,
и полная инструкция деплоя в README (раздел "Деплой" ниже, между "Первый вход в систему" и
"Профиль пользователя").

- **`vercel.json`** — `framework: vite`, явный `rewrites` (`/* → /index.html`) для client-side
  роутинга React Router: без него прямой переход или обновление страницы на любом пути, кроме
  `/`, вернули бы 404 от Vercel (он по умолчанию ищет файл на диске по пути запроса).
- **`railway.json`** — `startCommand` запускает `prisma migrate deploy` ПЕРЕД стартом сервера
  при каждом деплое: это и есть "прогон миграций Prisma в production" из Roadmap, а не только
  строчка в документации. Добавлен отдельный npm-скрипт `prisma:migrate:deploy` — существовавший
  `prisma:migrate` использует `migrate dev` (интерактивный, для разработки, может запросить
  сброс БД при конфликте) — использовать его в production было бы ошибкой.
- **`Procfile`** — альтернатива для Heroku-совместимых платформ, с явной `release`-фазой
  (миграции ровно один раз за деплой, а не при каждом рестарте, как в `railway.json`).
- **Найденное по ходу и вынесенное на видное место**: предупреждение про аватары на локальном
  диске backend-контейнера (нужен persistent volume на Railway) раньше было похоронено в
  середине README под "Профиль пользователя" — вынесено в сам раздел деплоя, где оно
  действительно нужно в момент настройки хостинга.
- **Смоук-тест** (Roadmap: "полный smoke-test в production... проверяются вручную") оформлен
  чек-листом — не может быть честно автоматизирован в этой среде, как и не может быть выполнен
  здесь: платформа нигде реально не развёрнута.

## Конфигурация окружений

Файлы строго по Roadmap: `.env.staging`, `.env.production` — шаблоны с пустыми значениями,
безопасные для коммита (тот же принцип, что и у уже существующего `.env.example`, Этап 0.2).

**Найденный при ревизии реальный блокер, а не формальность**: `NODE_ENV` валидировался через
`@IsEnum` с допустимыми значениями только `development`/`production`/`test` — значения
`staging` в списке не было вовсе. Если бы шаблон `.env.staging` с `NODE_ENV=staging` был
написан без этой проверки, приложение упало бы на валидации переменных окружения при самом
старте. Добавлено `Staging` в enum.

**Второй, более тонкий найденный блокер**: `isProduction()` (управляет `secure`/`sameSite`
cookie) проверял строго `nodeEnv === 'production'` — при `NODE_ENV=staging` cookie получила бы
`sameSite: 'lax'` вместо `'none'`, что **сломало бы аутентификацию** между staging-фронтендом
на Vercel и staging-бэкендом на Railway (разные домены, lax-cookie не отправляется
кросс-доменно). Staging разворачивается на той же кросс-доменной HTTPS-инфраструктуре, что и
production — исправлено: `isProduction()` теперь возвращает `true` и для `staging` тоже.

**Ревизия секретов по всему репозиторию** — проверено и подтверждено чистым:
- `.gitignore` корректно исключает `.env`/`.env.local` (реальные секреты), но не шаблоны.
- `JWT_ACCESS_SECRET`/`JWT_REFRESH_SECRET` обязательны (`@IsNotEmpty`, без дефолта в коде) —
  приложение не запустится без явного значения, а не тихо использует небезопасный дефолт.
- `BOOTSTRAP_ADMIN_PASSWORD` не имеет hardcoded fallback — при отсутствии `seed.ts` просто
  пропускает создание аккаунта с предупреждением, а не создаёт его со слабым паролем по умолчанию.
- Поиск паттернов хардкоженных API-ключей (`sk-...`, `AIza...`) по всему `backend/src` — ничего
  не найдено.

### Чек-лист секретов

| Секрет | Где взять | Обязателен | Уникален для каждого окружения |
|---|---|---|---|
| `DATABASE_URL` | Supabase/Railway Postgres | Да | Да — отдельная БД на окружение |
| `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET` | `openssl rand -base64 48` | Да | Да |
| `OPENAI_API_KEY` | platform.openai.com | Да (с Фазы 5) | Желательно (отдельный usage-лимит) |
| `GOOGLE_DRIVE_CLIENT_EMAIL`, `GOOGLE_DRIVE_PRIVATE_KEY` | Google Cloud Console, сервисный аккаунт | Да (с Этапа 4.2) | Рекомендуется |
| `QDRANT_URL`, `QDRANT_API_KEY` | Qdrant Cloud или self-hosted | Да (с Фазы 5) | Рекомендуется |
| `SENTRY_DSN` | sentry.io | Опционален (Этап 12.1) | Рекомендуется отдельный проект/tag |
| `BOOTSTRAP_ADMIN_EMAIL`, `BOOTSTRAP_ADMIN_PASSWORD` | Задать самостоятельно | Да | Да — разные учётки на окружение |

Правила: (1) ни один секрет никогда не коммитится — только пустые шаблоны; (2) реальные
значения задаются через переменные окружения хостинга (Railway/Vercel Dashboard), не
редактированием файлов в репозитории; (3) JWT-секреты и bootstrap-учётка не переиспользуются
между окружениями; (4) при компрометации — немедленная ротация и передеплой.

Frontend не получил отдельных `.env.staging`/`.env.production` — Roadmap называет их только для
backend, и `VITE_API_BASE_URL` (единственная переменная, не секрет — публичный URL) естественнее
задавать через встроенный per-environment UI Vercel, а не коммитить в репозиторий.


## E2E критичных сценариев

Файлы строго по Roadmap: `e2e/auth.spec.ts`, `e2e/application-flow.spec.ts`.

**То же честное ограничение среды, что и в Этапе 13.1**: `npm run test:e2e` в этой песочнице
не запускался — сетевая политика блокирует установку npm-пакетов. Дополнительно к этому e2e-тесты
(в отличие от unit) требуют **настроенной тестовой базы данных** (`DATABASE_URL`) — их нельзя
запустить вообще без реального Postgres, даже если бы установка пакетов работала.

- **`e2e/utils/create-test-app.util.ts`** — общая утилита поднятия РЕАЛЬНОГО приложения (общая
  для обоих spec-файлов, не задублирована): реальные guards/pipes/interceptors, реальная БД
  через Prisma — в этом смысл e2e, в отличие от unit-тестов (Этап 13.1), где всё замокано.
  Единственное, что подменяется — `GoogleDriveService`/`OpenAiService` через
  `overrideProvider()`: реальные вызовы требуют платных внешних credentials (Google Service
  Account, OpenAI API key) и сделали бы тесты медленными/дорогими/недетерминированными —
  стандартная практика для e2e с платными внешними API, а не ослабление теста.
- **`auth.spec.ts`** — логин с верными/неверными данными, защита от user enumeration (та же
  generic-ошибка для несуществующего email и неверного пароля) уже через реальный HTTP, не мок;
  полный цикл login → `/auth/me` с cookie → logout.
- **`application-flow.spec.ts`** — полный User Journey (Часть 6, п.17 ТЗ) одним непрерывным
  сценарием: создание заявки → чек-лист авто-генерируется из требований программы → загрузка
  документа → привязка к заявке → AI-чат сохраняет обмен сообщениями → чек-лист остаётся
  согласован. Сама фикстура (университет, программа, требование, цикл поступления, тип
  документа, тестовый пользователь) создаётся и подчищается в `beforeAll`/`afterAll` —
  `Checklist`/`ChecklistItem`/`ApplicationDocument`/`Chat`/`Message` подчищать отдельно не нужно
  (каскадное удаление вместе с `Application` в схеме).

Пользователю нужно самостоятельно настроить тестовую БД и выполнить
`npm install && npm run test:e2e` после скачивания.

Файлы строго по Roadmap: `auth.service.spec.ts`, `prompt-builder.service.spec.ts`,

API Jest (`describe`/`it`/`expect`/`jest.fn`/`jest.mock`), вычитаны и типо-проверены через

- Добавлена сама инфраструктура Jest — `package.json` уже содержал скрипты
  `test`/`test:watch`/`test:cov` (стандартный NestJS-скаффолд, Этап 0.1), но не сам блок
  конфигурации `"jest": {...}` — добавлен.
- **`AuthService`** — защита от user enumeration (одинаковая generic-ошибка для "нет email" и
  "неверный пароль"), инвалидация сессий через `tokenVersion`, деактивированный пользователь
  отклоняется даже с верным паролем.
- **`PromptBuilderService`** — честность при пустой базе знаний (edge-кейс, явно названный
  Roadmap), и оба лимита контекста из Этапа 12.3 (per-message и совокупный бюджет истории).
- **`ProgressService`** — edge-кейс "0 документов" (явно назван Roadmap), и что отсутствие
  требования (нет чек-листа, нет событий календаря) даёт 100%, а не 0% — не наказывает заявку.
- **`VerificationService`** — все три ветки анализа (изображение/PDF/DOC-DOCX), включая

артефакт этой песочницы (тот же класс проблемы, что с `process`/`Buffer` из `@types/node` с

Решение — явный `"types": ["node", "jest"]` в `tsconfig.json`, безопасный и в реальном проекте.


## AI Cost Optimization

Доработан именно `prompt-builder.service.ts`, как и требовал Roadmap.

- **Ограничение длины контекста, двухшаговое**: 1) каждое сообщение истории обрезается по
  отдельности (`MAX_HISTORY_MESSAGE_CHARS=2000`) — защита от одного очень длинного сообщения;
  2) совокупный объём истории ограничивается (`MAX_TOTAL_HISTORY_CHARS=8000`), отбрасывая
  САМЫЕ СТАРЫЕ сообщения при превышении бюджета — недавний контекст важнее для связности ответа.
  Найдено при ревизии: `AiService.HISTORY_LIMIT=20` ограничивал только КОЛИЧЕСТВО сообщений, не
  их длину — 20 сообщений по 10 000 символов (максимум по `SendMessageDto`) теоретически давали
  бы до 200 000 символов на один запрос. Результаты поиска по базе знаний тоже обрезаются
  (`MAX_KNOWLEDGE_RESULT_CHARS=1500`) — защита на случай изменения логики чанкинга в будущем.
- **Кэширование частых вопросов — осознанно НЕ реализовано** (Roadmap помечает это "если
  применимо"): финальный ответ AI глубоко персонализирован — включает имя студента, конкретный
  университет/программу, документы и этап именно ЕГО заявки. Кэш по тексту вопроса неизбежно
  отдал бы одному студенту персональный контекст другого при формально похожем вопросе — это
  баг корректности, а не оптимизация. Более узкое и безопасное кэширование (результатов
  embedding-поиска, university-scoped, не студенческое) технически применимо, но живёт в
  `EmbeddingsService`/`AiService`, а не в файле, на который явно ограничен этот этап.
- **Проверка (реальный OpenAI недоступен в этой среде)**: символьная симуляция синтетического
  худшего случая (20 сообщений истории по ~5000 символов + 5 результатов поиска по ~3000
  символов) показала сокращение примерно на 86% (с ~110 000 до ~15 500 символов, что при грубой
  оценке ~4 символа/токен — с ~27 500 до ~3 900 токенов). Символы — честная замена токенам за
  неимением реального токенизатора, а не точный подсчёт.


## Производительность

- **Документы** — добавлена настоящая пагинация (`page`/`limit`), которой не было вовсе с Этапа

- **Сообщения чата** — уже использовали осознанный паттерн "последние N" вместо page-based
  (документировано с явной ссылкой на ТЗ про "миллионы сообщений", естественнее для UX чата).
  Но при 1000+ сообщениях не было способа увидеть историю старше первых 200 (максимальный
  `limit`) — добавлен курсор `before` (Prisma cursor-пагинация) и кнопка "Показать более ранние

- **Lazy loading страниц** (`frontend/src/App.tsx`) — все 12 маршрутизируемых страниц переведены
  на `React.lazy` + единый `Suspense` вокруг `<Routes>`; каждая страница теперь отдельный
  JS-чанк, загружаемый только при переходе на неё, а не единым бандлом при первой загрузке.
- **Ревизия индексов БД — нашла реальный пробел**: `Document.status` не был проиндексирован,
  хотя именно по нему идёт глобальный запрос очереди `NeedsReview` (`DocumentsReviewService`,
  Этап 11.4) — без scoping по пользователю, то есть полный скан таблицы, растущий линейно с
  числом документов на всей платформе. Добавлен `@@index([status])`.


## Security hardening (настоящий Этап 12.1)

**Важное уточнение**: секция «Production hardening» ниже была изначально ошибочно названа
«Этап 12.1» — по факту это полезная, но самостоятельная работа без номера в 14-фазном плане
(Sentry, compression, graceful shutdown). Настоящий Этап 12.1 — это следующее:

- **`backend/src/common/guards/rate-limit.guard.ts`** — кастомный guard поверх встроенного
  `ThrottlerGuard`, трекает лимит по `userId` для аутентифицированных запросов вместо IP
  (стандартный `ThrottlerGuard` трекает по IP — несколько пользователей за одним IP делили бы
  лимит, а один пользователь мог бы обойти его сменой сети). Зарегистрирован глобально вместо
  обычного `ThrottlerGuard` — все существующие `@Throttle`-декораторы (login, refresh, AI-чат,
  AI-рекомендации, загрузка документов) автоматически используют новую трекинг-логику без
  изменения самих декораторов.
- **XSS** — проверено: во всём frontend нет ни одного `dangerouslySetInnerHTML`, весь
  пользовательский контент (заметки, комментарии) рендерится через `{}` в JSX, React
  экранирует автоматически.
- **CSRF** — `SameSite=None` в production (нужен для cross-origin Vercel↔Railway, Этап 1.1)
  митигирован строгим CORS (конкретный `corsOrigin`, не wildcard, + `credentials: true`):
  браузер блокирует state-changing запросы с чужого origin на этапе preflight.
- **Финальный security-review всех эндпоинтов — нашёл реальный пропущенный пробел**:
  `GET /knowledge` и `GET /knowledge/:id` оставались `@Public()` — та же причина для

## Production hardening

Начал с инвентаризации: Helmet, глобальный `ThrottlerModule`, `HttpExceptionFilter` без утечки
стектрейсов, health-check с реальной проверкой БД — всё это уже было с ранних этапов (0.2, 1.1).
Не стал дублировать готовое — сосредоточился на подтверждённых пробелах:

- **Error tracking (Sentry)** — новый `ErrorTrackingService`, опционален (без `SENTRY_DSN`
  работает как раньше, ошибки только в серверном логе — тот же принцип "не ломает работу из
  коробки", что и у AiPrompts). Захватывает только 5xx — ожидаемые 4xx (валидация,
  403 и т.п.) не засоряют Sentry шумом. Потребовало перевести `HttpExceptionFilter` с ручного
  `new HttpExceptionFilter()` в `main.ts` на DI-регистрацию через `APP_FILTER` (`AppModule`) —
  иначе внедрить `ErrorTrackingService` в конструктор фильтра было бы невозможно.
- **`enableShutdownHooks()` — найденный и исправленный реальный пробел**: `PrismaService` имеет
  `onModuleDestroy` ещё с Этапа 0.2, но без этого вызова Nest никогда не запускал бы этот хук
  при SIGTERM — соединение с БД обрывалось бы резко при каждом деплое (Railway заменяет
  контейнер), а не закрывалось явно.
- **Compression** — не было вовсе, добавлено.
- **Явный лимит размера тела запроса** (2MB, с запасом относительно `KnowledgeBase.content`,
  крупнейшего JSON-поля в проекте) — раньше использовался недокументированный дефолт Express.
- **Throttle на `/auth/refresh` (20/мин) и `POST /documents/upload` (10/мин)** — раньше не были
  ограничены вовсе, хотя upload запускает Google Drive + асинхронную AI-проверку (дорогая
  операция).
- **CSRF — задокументировано, не добавлена новая инфраструктура**: `SameSite=None` (Этап 1.1,
  нужен для cross-origin Vercel↔Railway) сам по себе ослабляет встроенную защиту браузера, но
  уже митигирован строгим CORS (конкретный `corsOrigin`, не wildcard, + `credentials: true`) —
  браузер блокирует state-changing запросы с чужого origin на этапе preflight. Отдельный
  CSRF-токен был бы избыточен поверх уже действующей защиты.


## DocumentTypes / AiPrompts / Settings

*(Примечание: этот раздел изначально был ошибочно назван "Этап 11.3" — в оригинальном Roadmap
под 11.3 значится Admin: Knowledge Base, реализованный отдельно, см. выше. Работа, описанная
ниже, реальна и сделана, но не имеет отдельного номера этапа в 14-фазном плане.)*

Три разные по природе сущности, объединённые в одном этапе:

- **DocumentTypes** — получил полноценный CRUD (раньше было только чтение с Этапа 4.1).
  Добавлено `isActive` (soft-delete) — hard delete нарушил бы FK с `Document`/
  `ProgramRequirement`, тот же принцип, что у University/Program. **Нашёл и сразу исправил у
  себя повторение той же ошибки, что уже была найдена в Этапе 11.2**: изначально оставил

- **AiPrompts** — новая сущность из ТЗ ("Позволяет менять поведение ИИ без изменения кода").

- **Settings** — в схеме это `userId`-scoped (self-service), а не то, чем Admin управляет для
  других, поэтому реализовано как расширение уже существующей страницы `/settings` (Этап 1.4),
  а не под `/admin`: `GET/PUT /users/settings` (тема, язык, часовой пояс, email-уведомления),
  `upsert` гарантирует, что первый визит на страницу сразу показывает дефолты без отдельного
  шага создания.


## Admin: Universities/Programs/Requirements/Cycles

`/admin` (Admin/SuperAdmin) — три вкладки поверх уже готового с Фазы 2 backend CRUD:
Университеты, Программы (с вложенным управлением требованиями по клику на строку), Циклы
поступления. Формы через react-hook-form + zod, создание/редактирование в одной форме
(различается только наличием `editingId`).

**Существенная находка при подготовке, а не просто новый UI:** прежде чем писать
интерфейс деактивации, проверил — `GET /universities` (и аналогично Programs/AdmissionCycles)
всегда фильтровал `isActive: true` безусловно, даже для Admin. Это был **осознанно отложенный
пробел**: ещё в Этапе 2.1 в коде был явный комментарий "Возможность увидеть деактивированные
вузы понадобится в Admin Panel (Фаза 11) — сейчас намеренно не проектируется". Без исправления
Admin деактивировал бы университет через UI и тот немедленно исчезал бы из списка навсегда —
реактивировать стало бы невозможно.

**Решение потребовало более глубокой правки, чем ожидалось:** первая идея — оставить роут
`@Public()` и доставать текущего пользователя опционально — не сработала: `@Public()`
**полностью пропускает** выполнение JWT-стратегии (`if (isPublic) return true`, минуя
`super.canActivate`), поэтому `req.user` не заполняется даже для залогиненного Admin. Убрал
`@Public()` с чтения (`GET`) у всех трёх ресурсов — платформа закрытая, весь frontend и так
работает только под авторизацией, анонимный доступ никогда не был практически нужен — и добавил
query-параметр `includeInactive`, который сервис проверяет на стороне backend по роли текущего
пользователя, а не доверяет клиенту напрямую.


## Admin: Users management

Единственный способ появления нового пользователя на платформе — самостоятельная регистрация
не предусмотрена (ТЗ). Весь контроллер под `RolesGuard(Admin, SuperAdmin)` — Mentor намеренно
**не** имеет доступа, в отличие от `GLOBAL_READ_ROLES`-доступа к заявкам: управление
пользователями более чувствительная операция.

- `GET /admin/users` — список (пагинация, фильтры `role`/`isActive`, поиск по email/имени).
- `POST /admin/users` — создание (email, пароль, имя, роль).
- `PUT /admin/users/:id/role` — смена роли.
- `PUT /admin/users/:id/deactivate`, `PUT /admin/users/:id/activate` — **деактивация, не
  удаление** ("Удаление пользователя запрещено", ТЗ). При деактивации и при смене роли
  `tokenVersion` инкрементируется — уже выданные токены немедленно теряют силу, не дожидаясь
  истечения (`JwtAccessStrategy` проверяет `isActive` на каждый запрос ещё с Этапа 1.1 — эта
  инфраструктура уже была на месте, здесь она наконец используется по назначению).
- **Защита от эскалации привилегий**: только SuperAdmin может выдать роль SuperAdmin — Admin
  не может назначить эту роль ни себе, ни другим.

**Устранено найденное по ходу дублирование**: константа раундов bcrypt существовала только в
`UsersService` (Этап 1.3); теперь нужна и здесь — вынесена в общую `common/utils/password.util.ts`,
`UsersService` тоже переведён на неё.

## RBAC и Policy для Mentor

Не аудит, а закрытие реальных пробелов между решением раунда 3 ("Mentor может: просматривать
все заявки; оставлять комментарии; отмечать ChecklistItems; изменять статус документов;
добавлять и редактировать события календаря; создавать заметки") и тем, что было фактически
реализовано на момент начала этого этапа. Три пункта из решения не имели вообще никакой
реализации:

- **`MentorComments`** — отдельная сущность с `isInternal`, архитектурно решённая с самого
  начала проекта, реализована только сейчас. `POST/GET /mentor-comments/...`,
  `PUT/DELETE /mentor-comments/:id`. В отличие от Notes (пишут и студент, и наставник) —
  создаёт только Mentor/Admin/SuperAdmin, это обратная связь ОТ наставника. `applicationId`
  ИЛИ `documentId` (хотя бы одно) — проверяется на уровне сервиса, Prisma schema DSL не
  выражает декларативно "хотя бы одно из двух" без сырой SQL CHECK-миграции.
- **`PUT /checklists/items/:id`** — ручная отметка пункта чек-листа. До этого единственным

  автоматической AI-проверки (Этап 6.2), иначе ручное одобрение вело бы себя иначе, чем
  автоматическое.

**Формализация уже реализованного:** проверка финальных статусов заявки (Этап 3.1) вынесена
из приватного безымянного метода `ApplicationsService` в отдельный класс
`ApplicationStatusPolicy` — та же логика, но с собственной точкой в коде, соответствующей
названию из планирования.

Frontend в этом этапе не затронут — Roadmap называет его backend-этапом.


## ProgressCalculationService

Веса (Documents 35%, Essays 25%, Checklist 20%, Deadlines 10%, Interview 10%) и решение "не
хранить прогресс в БД" — архитектурные решения, принятые ещё на этапе планирования, до начала
разработки. `GET /progress/:applicationId` всегда пересчитывает заново из актуального
состояния чек-листа/документов/календаря — не может "отстать" от реальности, но и не кэшируется
(цена — 5 запросов к БД на каждый расчёт, приемлемо для объёма данных одной заявки).

**Честная оговорка по двум из пяти компонентов:** в схеме нет отдельных сущностей "Essay" или
"Interview" — это интерпретации поверх существующих данных, а не факты из ТЗ:
- **Essays** — доля выполненных пунктов чек-листа, чей `documentType.name` — "Essay" или
  "Motivation Letter" (единственные подходящие типы из засеянного справочника, Этап 2.2).
- **Interview** — статус (`completed`) самого позднего события календаря заявки с
  `type: Interview`, если оно есть.

Если для компонента нет применимых данных (например, программа не требует эссе, интервью ещё
не назначено) — засчитывается 100%, а не 0%: заявка не должна штрафоваться за отсутствие
требования, которого у неё нет.

Frontend — полоса прогресса и разбивка по 5 компонентам в начале `OverviewTab`.


## AiSuggestions — гибридная модель

Как и `ActivityLog`, этой таблицы нет в исходном ТЗ — "гибридная модель AI-рекомендаций
(динамические простые проверки + сохранённые долгосрочные рекомендации)" была принята как
самостоятельное архитектурное решение ещё на этапе планирования, до начала разработки.

- **Динамические** — вычисляются заново при каждом запросе из уже существующих данных
  (незавершённые пункты чек-листа, дедлайн ближе 14 дней) и нигде не хранятся: не могут
  "устареть", в отличие от сохранённых.
- **Сохранённые** (`AiSuggestion`) — требуют вызова OpenAI (дорогая операция), поэтому
  генерируются только по явному запросу (`POST /ai-suggestions/:applicationId/generate`,
  строгий throttle — тот же принцип, что у `/ai/chat`, Этап 5.5) и кэшируются в БД. Можно
  скрыть (`PUT /ai-suggestions/:id/dismiss`, soft — `isDismissed`), крестик закрытия на
  frontend показан только для сохранённых — динамические исчезают сами, когда условие
  перестаёт быть актуальным.
- Владелец заявки или Mentor/Admin/SuperAdmin — тот же паттерн доступа, что и у Notes/Chat/
  ActivityLog.
- Frontend — блок «AI-рекомендации» на `OverviewTab`, рядом с «Последние действия» (Этап 8.2).


## ActivityLog

**В исходном ТЗ таблицы `ActivityLog` нет вовсе** — журнал действий описан только нарративно
в Части 6 ("Каждая заявка хранит журнал активности", примеры: "Uploaded Passport", "Started
Harvard Chat"), без структуры полей. Схема спроектирована самостоятельно, отталкиваясь от этих
примеров и требования Roadmap подписаться на события четырёх модулей.

- `GET /activity-log/:applicationId` — только чтение (владелец заявки или Mentor/Admin/
  SuperAdmin, тот же паттерн, что у Notes/Chat/Checklist). Создание — только внутренний метод
  `ActivityLogService.log()`, вызывается другими сервисами.
- `ActivityLogModule` намеренно держится "листом" в графе зависимостей (только `PrismaService`,
  ничего не импортирует сам) — тот же принцип, что уже применён к `GoogleDriveModule`/
  `OpenAiModule`: позволяет подключить его сразу к 4 модулям (Documents/Checklists/Chat/
  Calendar) без циклических зависимостей.
- **Разная стратегия отказоустойчивости в зависимости от контекста**: логирование внутри
  транзакции создания заявки (авто-привязка документов) НЕ обёрнуто в try/catch — сбой должен
  откатить всю транзакцию, как и генерация чек-листа; логирование вне транзакций (ручная
  привязка документа, отправка сообщения, событие календаря) обёрнуто — сбой лога не должен
  ломать уже совершённое основное действие.
- Frontend — блок "Последние действия" на `OverviewTab` (вместо восьмой вкладки в Workspace,
  которой нет в структуре ТЗ) — соответствует альтернативе, изначально заложенной в Roadmap
  ("или блок на Overview").


## Notes API + Frontend

**Закрыт открытый вопрос**, оставленный на этапе архитектурных решений (раунд 3, §6, п.1):
"заметки Mentor должны быть по умолчанию не видны студенту?" — добавлено поле `isInternal` по
аналогии с уже принятым `MentorComments.isInternal` (тот же паттерн видимости: `false` видят
все с доступом к заявке, `true` — только Mentor/Admin/SuperAdmin).

- `POST/GET /notes/:applicationId`, `PUT/DELETE /notes/:id` — создание доступно владельцу
  заявки или Mentor/Admin/SuperAdmin; редактирование и закрепление — только автору, независимо
  от роли; удаление (soft-delete через `deletedAt`) — автор или Admin/SuperAdmin, **не**
  Mentor (тот же принцип, что и в Calendar: Mentor нигде не получал прав на удаление).
- Student не может пометить свою заметку как `isInternal=true` — бессмысленно скрывать что-то
  от самого себя, при попытке значение принудительно игнорируется на backend.
- Frontend (`NotesTab`) — создание, инлайн-редактирование, закрепление, удаление; чекбокс
  "только для наставников" виден только пользователям с соответствующей ролью; кнопки
  редактирования/удаления видны только автору заметки.


## Calendar + Notifications (Frontend)

`/calendar` — общий календарь (все свои события), `CalendarTab` — события конкретной заявки.
Оба используют один и тот же `CalendarEventForm` (различается только переданный

счётчик непрочитанных, отметка прочитанным по клику и «прочитать все»; закрывается по клику
вне панели (новый переиспользуемый хук `useClickOutside`).

**Важная деталь процесса:** часть этого этапа (типы, константы меток, `calendarApi.ts`) уже
существовала в рабочей директории на момент начала — тот же эффект, что и в Этапах 1.4 и 7.1.
Провёл аудит перед тем, как строить поверх: код оказался корректным и даже удачнее моего
собственного плана — общий файл констант меток типов/приоритетов, которого я сам не планировал
заранее, а собирался дублировать в каждом компоненте.


## Cron job дедлайнов

Ежедневно (`@Cron(CronExpression.EVERY_DAY_AT_8AM)`) проверяет все незавершённые будущие
события календаря на приближение дедлайна (30/7/3/1 дней) и создаёт `Notification`
(`type: Calendar`) через уже существующий внутренний `NotificationsService.create` (Этап 7.2).

**Защита от дублей** — через поле `notifiedStages` (заложено в схему ещё на Этапе 7.1
специально под эту задачу): результат сохраняется в БД **после каждой отдельно отправленной
стадии**, а не одним махом в конце обработки события — если между стадиями 7 и 3 произойдёт
сбой, уже отправленная стадия 7 не продублируется на следующем запуске крона. Сбой по одному
событию логируется и не прерывает обработку остальных событий в этом же запуске.

Если событие создано, когда до дедлайна уже осталось немного (например, 5 дней), несколько
стадий (30 и 7) окажутся "просроченными" одновременно на первом же запуске — это ожидаемое
поведение, а не баг: пользователь получает оба актуальных напоминания сразу, а не теряет их.


## Notifications API

- `GET /notifications` — список своих уведомлений (пагинация, фильтры `type`/`isRead`).
- `PUT /notifications/:id/read`, `PUT /notifications/read-all` — отметка прочитанным.
- **Нет `POST`** — создание уведомлений (`NotificationsService.create`) только внутренний
  метод, вызывается другими backend-сервисами (например, будущим cron дедлайнов, Этап 7.3).
  Все 6 типов (System/AI/Calendar/Document/University/Admin) подразумевают системную
  генерацию — у пользователя нет сценария "создать уведомление самому себе".


## Calendar API

`POST/GET/PUT/DELETE /calendar[/:id]` — `applicationId` опционален: без него личное событие
(строго приватное, доступ только владельцу, даже для Mentor/Admin), с ним — событие заявки.
`GET /calendar` без `applicationId` — только свои события (личные + по своим заявкам), не
"все события всех пользователей" даже для ролей с глобальным доступом; с `applicationId` —
события конкретной заявки, доступной владельцу или Mentor/Admin/SuperAdmin.

**Важное уточнение прав Mentor, найденное при аудите:** для событий заявки Mentor может
создавать и редактировать (решение по правам Mentor: "может добавлять и редактировать события
календаря"), но **не удалять** — в том решении удаление не упоминалось ни для одного ресурса
(ни чек-листа, ни документов, ни календаря). Удаление события заявки — только владелец или
Admin/SuperAdmin.

Значительная часть этого этапа (схема, DTO, сервис, контроллер) уже существовала в рабочей
директории на момент начала работы — судя по всему, от прерванного ранее прохода (тот же
эффект, что и в Этапе 1.4). Провёл полный построчный аудит вместо слепого доверия и нашёл
именно вышеописанное несоответствие в правах на удаление, которое исправил.


## StatusBadge (Frontend)

Вынесен из `DocumentRow` в переиспользуемый компонент — логика меток/цветов статуса документа
(`Uploaded/Checking/Approved/Rejected/NeedsReview/Expired`) раньше жила прямо в `DocumentRow`,
теперь отдельно и переиспользуема. `DocumentRow` дополнительно показывает объяснение AI-проверки
(`document.verificationResult`) второй строкой под именем файла — только когда оно есть.

**Попутно закрыт пробел на backend:** `ApplicationDocumentResponse.document` (Этап 4.3) не
отдавал `verificationResult`, хотя это поле в `Document` существует с Этапа 4.1 — без него
объяснение AI-проверки нечем было бы показать. Добавлено в интерфейс и маппинг.


## Авто-отметка чек-листа

`ChecklistsService.markItemsCompletedForApprovedDocument()` вызывается из `VerificationService`
(Этап 6.1) сразу после простановки документу статуса `Approved`. Документ может быть привязан
сразу к нескольким заявкам (переиспользуемые документы вроде паспорта — `ApplicationDocuments`,
Этап 4.3) — метод находит ВСЕ такие заявки через связи и отмечает соответствующие пункты во
всех чек-листах, а не только в одном.

**Найденная и исправленная по ходу проблема с порядком try/catch:** синхронизация чек-листа
изначально была внутри общего `try` блока `verifyDocument()` — при её сбое ошибка попадала бы
во внешний `catch` и **перезаписывала уже корректно определённый статус `Approved` на
`NeedsReview`**, хотя сама проверка документа прошла успешно и ошибочным был только вторичный
побочный эффект. Вынес синхронизацию в отдельный `try/catch`: её сбой теперь только логируется,
не искажая результат основной проверки.

Обновление идемпотентно (`WHERE isCompleted: false` в условии `updateMany`) — повторный вызов
(например, при будущей ре-верификации) не перезатирает уже существующие `completedAt`/
`completedBy` у пунктов, отмеченных ранее.


## Document Verification Engine

Запускается автоматически после успешной загрузки документа в Google Drive (`DocumentsService.upload`,
Этап 4.2) — **асинхронно (fire-and-forget)**, не блокирует ответ на запрос загрузки: vision-анализ
изображения может занять несколько секунд, и ждать его в одном HTTP-запросе с загрузкой файла было
бы плохим UX.

- **Изображения** (JPEG/PNG/WEBP) — анализируются напрямую через vision-запрос к OpenAI
  (`OpenAiService.analyzeDocumentImage`, новый метод, намеренно отдельный от текстового
  `createChatCompletion` — больше никто в проекте не отправляет изображения в OpenAI).
- **PDF** — текст извлекается через `pdf-parse`, затем анализируется как обычный текст.
- **DOC/DOCX** — честное ограничение: автоматический анализ не поддерживается, статус сразу
  `NeedsReview`. Надёжный парсинг старого формата Word требует отдельной специализированной
  библиотеки, которую нельзя проверить в этой среде без сети — не стал притворяться, что
  анализ произошёл.
- Результат — `Approved`/`Rejected`/`NeedsReview` с текстовым объяснением на русском. Любая
  внутренняя ошибка (сбой OpenAI, Drive, парсинга ответа) перехватывается и тоже приводит к
  `NeedsReview` с объяснением — документ никогда не остаётся забытым в статусе `Checking`.

**Честная оговорка, как и с остальными внешними интеграциями:** живая проверка в этой среде
невозможна — ни сети, ни credentials.


## AiSessions

Без контроллера (как Google Drive/OpenAI/Embeddings) — используется только внутренне,
подключено прямо в `AiService.chat()` (Этап 5.5): вокруг вызова OpenAI сессия находится или
создаётся (find-or-create), после ответа в неё пишутся суммарные токены (prompt + completion).
Сбой учёта токенов не блокирует сохранение ответа пользователю — только логируется.

**"Одна активная сессия на чат"** (ТЗ, Часть 2, п.20) реализована на уровне сервиса
(find-or-create: если активная сессия уже есть — переиспользуется, а не создаётся вторая), а
не через partial unique index в БД — `UNIQUE WHERE endedAt IS NULL` не выражается декларативно
в текущей версии Prisma schema DSL без сырой SQL-миграции, которую в этой среде всё равно
нельзя было бы прогнать и проверить.

`endActiveSession()` реализован, но пока нигде не вызывается автоматически — нет ни таймаута
неактивности, ни явного триггера от frontend в текущем объёме проекта; готовая точка входа для
будущего использования.


## AI Chat API

- `POST /ai/chat` — единственный вход в RAG-пайплайн (Часть 3, §11 ТЗ): заявка → университет →
  история → поиск по базе знаний → документы студента → Prompt Builder → OpenAI → атомарное
  сохранение вопроса и ответа → возврат. Только владелец заявки (не Mentor — ассистент отвечает
  конкретному студенту, наставник не инициирует диалог от его лица). Rate limiting строже
  общего (10/мин) — обещано ещё в `main.ts` (Этап 1.1).
- **Отклонение от буквального ТЗ:** тело запроса — `{applicationId, message}`, не
  `{chatId, message}` — Chat 1:1 с Application (Этап 5.2), весь frontend-контекст уже построен
  вокруг `applicationId`, заводить второй идентификатор для того же самого было бы избыточно.
- **Prompt Builder** строго по структуре ТЗ (System Prompt → студент → университет/программа/
  этап → документы → результаты поиска по базе знаний → сообщение), с честным отказом,
  зашитым в system prompt при пустых результатах поиска. Это указание модели через промпт, а не
  программная гарантия — полностью исключить "выдумывание" ответа кодом невозможно.
- Вопрос и ответ сохраняются одной транзакцией (`ChatService.saveExchange`) — сбой между двумя
  отдельными записями не может оставить в истории вопрос без ответа.

**Честная оговорка, как и с Google Drive/Qdrant:** живая проверка полного RAG-пайплайна в этой
среде невозможна — ни сети, ни credentials OpenAI/Qdrant.


## Embedding Service

Без контроллера (как GoogleDriveModule/OpenAiModule) — используется только внутренне:
`KnowledgeService` вызывает индексацию при создании/обновлении статьи и удаление векторов при
удалении; поиск (`EmbeddingsService.search`) пока не вызывается снаружи — станет нужен Prompt
Builder (Этап 5.5).

- **Чанкинг** — по границам абзацев с перекрытием (~1000 символов, приближение по символам без
  токенайзера), принудительное разбиение для аномально длинных абзацев.
- **Embeddings** — `OpenAiService.createEmbedding()` (новый метод, та же изоляция SDK, что и
  для чата), модель `text-embedding-3-small` по умолчанию (`OPENAI_EMBEDDING_MODEL`).
- **Переиндексация без хранения ID точек в Postgres**: у статьи несколько чанков (векторов), а
  не один — вместо хранения списка ID точек в БД, удаление старых векторов при переиндексации
  и при удалении статьи выполняется через фильтр Qdrant по `payload.articleId`.
  `KnowledgeBase.embeddingId` теперь не буквальный ID точки, а маркер "статья проиндексирована"
  — реальный источник истины по векторам одной статьи — сам Qdrant.
- **Индексация не блокирует сохранение статьи**: если Qdrant/OpenAI временно недоступны,
  статья всё равно сохраняется в Postgres (`isIndexed: false`). Отличается от подхода к
  документам (Этап 4.2, где сбой Google Drive блокирует загрузку целиком) — там смысл запроса
  именно "положить файл в Drive", здесь — "сохранить текст", индексация вторична.

**Честная оговорка:** живая проверка против реального Qdrant/OpenAI в этой среде невозможна —
ни сети, ни credentials. Код написан по официальным клиентам (`@qdrant/js-client-rest`, `openai`),
проверка — на твоей стороне.


## Knowledge Base API

- `GET /knowledge`, `GET /knowledge/:id` — публично (пагинация, фильтры `universityId`/
  `category`, поиск `search` по заголовку). 11 категорий из ТЗ (Admission, Scholarships,
  Housing, Visa, Requirements, Programs, Deadlines, FinancialAid, FAQ, Contacts, Dormitory).
- `POST/PUT/DELETE /knowledge/:id` — только Admin/SuperAdmin.
- **Отклонение от паттерна soft-delete** (University/Program/AdmissionCycle/Application):
  `DELETE` — настоящее удаление, не `isActive=false`. На `KnowledgeBase` ничего не ссылается
  по внешнему ключу, поэтому риска нарушить целостность нет, а Roadmap явно просит
  протестировать именно удаление статьи.
- `embeddingId` (ID вектора в Qdrant) не отдаётся в API напрямую — вместо этого производное
  `isIndexed: boolean`. Индексация в Qdrant — Этап 5.4, не этот: та же последовательность,
  что Documents (4.1) → Google Drive (4.2).


## Chat API

- `GET /chat/:applicationId?limit=50` — история сообщений (владелец заявки или
  Mentor/Admin/SuperAdmin). Не полная история, а последние `limit` сообщений (по умолчанию 50,
  максимум 200) — ТЗ (Часть 2, п.23) прямо предупреждает про масштаб "миллионы сообщений".
- `POST /chat/message` — отправка (`applicationId` в теле, как буквально указано в Roadmap, без
  `:id` в пути). Пока без AI-ответа — сохраняется только сообщение пользователя; AI-ответ
  появится в Этапе 5.5 (Prompt Builder). Строже, чем чтение: писать может только владелец
  заявки — Mentor видит переписку, но не пишет в неё от лица студента.
- Удаления истории **нет и не будет**: `DELETE /chat/history` из исходного списка эндпоинтов
  ТЗ прямо противоречит правилу того же документа ("Запрещается удалять историю сообщений",
  Часть 2, п.8) — не реализован сознательно.
- Чат создаётся автоматически и атомарно при создании заявки (`POST /applications`, внутри той
  же транзакции, что и чек-лист и авто-привязка документов) — 1:1 с заявкой, не с университетом,
  как в исходной версии схемы (актуализировано архитектурными решениями до начала разработки).


## OpenAI Service

Изолированный адаптер к OpenAI API — без бизнес-логики (ни истории сообщений, ни поиска по
базе знаний, ни системного промпта — это Prompt Builder, Этап 5.5) и без контроллера, по
аналогии с `GoogleDriveModule` (Этап 4.2): "Frontend никогда не работает с OpenAI напрямую".
Ошибки API преобразуются в понятные исключения: 429 → `ServiceUnavailableException` (лимит),
таймаут → `RequestTimeoutException`, 401/403 → `InternalServerErrorException` (не раскрывая
детали проблемы с ключом пользователю).

**Честная оговорка:** живой запрос к OpenAI в этой среде проверить нельзя — нет ни сети, ни
API-ключа. Код написан по официальному SDK (`openai`), но проверка возможна только на твоей
стороне после `npm install` и заполнения `OPENAI_API_KEY` в `.env`.


## Documents tab (Frontend)

`UploadCard` — выбор вида документа + файла, клиентская валидация MIME/размера до отправки
(дублирует серверную из Этапа 4.1 — мгновенная обратная связь без ожидания ответа сервера).
Загрузка для конкретной заявки — на самом деле два backend-вызова (`POST /documents/upload` →
`POST /application-documents`), объединённые в одну фронтенд-мутацию, чтобы для пользователя
это выглядело единым действием.

`DocumentRow` — список в стиле GitHub Files (статус, индикатор "Общий" для isShared=true,
ссылка "Открыть" на Drive, отвязка). Первое реальное использование `react-icons` в проекте —
он был в стеке с самого начала, но не использовался; иконки уместнее эмодзи для монохромного
Swiss-дизайна (Часть 4 ТЗ).

Ручной выбор "привязать уже загруженный документ к другой заявке" не реализован в этом
этапе — Roadmap просил загрузку + список + индикатор "общий/уникальный", а не полноценный
picker существующих документов; сам механизм переиспользования (Этап 4.3) уже работает
автоматически при создании новой заявки.


## ApplicationDocuments API

- `POST /application-documents` — привязка существующего документа к заявке (`isShared`).
- `GET /application-documents/:applicationId` — список документов заявки.
- `DELETE /application-documents/:id` — отвязка: удаляет только связь, физический документ
  (запись `Document`, файл в Google Drive) не затрагивается — он может быть привязан к другим
  заявкам.

**Наконец реализована точка расширения, оставленная в `ChecklistsService` на Этапе 3.2**
("авто-отметка по уже одобренным shared-документам" — тогда `Documents` ещё не существовал).
Теперь при создании заявки `ApplicationsService` в одной транзакции: запрашивает
`ProgramRequirements` один раз → находит уже одобренные документы пользователя по типу →
авто-привязывает их к новой заявке (`ApplicationDocumentsService`) → передаёт тот же список
требований в `ChecklistsService`, который отмечает соответствующие пункты выполненными.
Сигнатура `ChecklistsService.generateForApplication()` изменилась: теперь принимает уже
готовые `requirements` и набор `autoCompletedDocumentTypeIds` от оркестратора, а не вычисляет
их сама — иначе оба сервиса (Checklists и ApplicationDocuments) дублировали бы один и тот же
запрос `ProgramRequirements`.


## Google Drive Service

Загрузка документа (`POST /documents/upload`) теперь выполняет полный пайплайн из ТЗ синхронно
в рамках одного запроса: MIME/размер (Этап 4.1) → перенос в Google Drive → запись `driveUrl`/
`driveFileId`/`driveFolderId` → удаление временного файла. `DocumentResponse.driveUrl` теперь
реальная ссылка на файл.

Структура папок в Drive: `Students/{userId}/{DocumentType.name}/`. **Отклонение от буквального
списка в ТЗ (Часть 2, п.21):** там перечислено 10 названий папок, не совпадающих 1:1 с уже
засеянными 14 `DocumentType` (Часть 2, п.11) — папки создаются динамически по реальным записям
`DocumentType`, а не по отдельному хардкод-списку, чтобы не было двух независимых источников
истины (особенно с учётом, что Admin Panel в Фазе 11 сможет добавлять новые виды документов).

**Кэширование** — без него каждая загрузка стоила бы 3-4 запроса к Drive API только на поиск/
создание нужной папки: ID корневой папки пользователя кэшируется на `User.driveRootFolderId`,
ID подпапок по видам документов — в отдельной таблице `UserDriveFolder`. ID папки "Students"
кэшируется в памяти процесса (известное ограничение: при нескольких инстансах backend возможна
гонка на "холодном" старте — приемлемо для монолитного MVP-деплоя, см. комментарий в коде).

**Честная оговорка:** реальную загрузку в Google Drive в этой среде проверить нельзя — нет
доступа в интернет и реальных сервис-аккаунт credentials. Код написан по официальному API
(`googleapis`) и пайплайну из ТЗ, но живая проверка против настоящего Google Drive возможна
только на твоей стороне после `npm install` и заполнения `GOOGLE_DRIVE_CLIENT_EMAIL`/
`GOOGLE_DRIVE_PRIVATE_KEY` в `.env`.


## Documents / DocumentTypes API

Пайплайн из ТЗ (Часть 3, п.13): `NestJS → проверка MIME → проверка размера → (антивирус,
опционально) → Google Drive → запись ссылки → удаление временного файла`. Этот этап реализует
только первые два шага — Google Drive интеграция появится в Этапе 4.2.

- `POST /documents/upload` — `multipart/form-data` (поле `file` + `documentTypeId`), до 10MB,
  PDF/JPEG/PNG/WEBP/DOC/DOCX. Файл сохраняется во временную директорию backend
  (`/uploads/documents-temp`), `driveUrl`/`driveFileId` пока `null` — заполнятся в Этапе 4.2.
- `GET /documents` — список своих документов (без пагинации — это личные документы одного
  пользователя, не каталог из тысяч записей).
- `GET /document-types` — публичный справочник видов документов (Passport, IELTS, Essay и
  т.д.), предзаполненный через `prisma/seed.ts` ещё в Этапе 2.2 — здесь впервые появился
  REST-слой поверх уже существующей таблицы.

**Рефакторинг существующего кода:** конфигурация Multer для документов почти дословно повторяла
код `avatar-storage.config.ts` (Этап 1.3) — вынес общую фабрику
(`common/utils/multer-disk-storage.util.ts`) и переписал под неё оба конфига, включая уже
существующий аватарный.


## Application Workspace (Frontend)

`/applications/:id` — 7 вкладок (Обзор/AI/Документы/Чек-лист/Календарь/Заметки/Статус).
Реально работают три: Обзор (базовая информация), Чек-лист (данные из Этапа 3.2 + локальный
прогресс "выполнено X из Y" — это НЕ полный взвешенный прогресс Фазы 9, а простая метрика по
самому чек-листу) и Статус (реальная смена статуса через `PUT /applications/:id/status`,
финальные статусы скрыты из выбора для не-админов на уровне UI). Четыре вкладки
(AI/Документы/Календарь/Заметки) — честные заглушки со ссылкой на фазу, где появятся;
используют общий `ComingSoonNotice` вместо копирования разметки в 4 файлах.

`ApplicationCard` на Dashboard стал кликабельным — в Этапе 3.3 вести было ещё некуда.


## New Application Wizard (Frontend)

`/applications/new` — 4 шага (университет → программа → цикл поступления → подтверждение),
состояние поднято в родительский `NewApplicationWizard` (без Context/Zustand — самодостаточный
локальный флоу на 3 значения). Ошибка дубликата (409 от backend) обрабатывается отдельным
понятным сообщением, а не общим "Не удалось создать заявку".

**Честная оговорка:** после успешного создания вести пользователя пока некуда — Application
Workspace появится только в Этапе 3.4. Вместо мёртвой ссылки — редирект на Dashboard, который
в рамках этого же этапа получил реальный список заявок (`GET /applications`) вместо пустой
заглушки, так что результат создания сразу видно. Карточки заявок пока не кликабельны.


## Checklists API

- `GET /checklists/:applicationId` — чек-лист заявки (доступ — как у Applications: владелец
  или Mentor/Admin/SuperAdmin).
- Генерация происходит автоматически и атомарно внутри `POST /applications`: копирование
  `ProgramRequirements` → `ChecklistItems` выполняется в одной Prisma-транзакции с созданием
  самой заявки (не постфактум-хуком) — если генерация упадёт, заявка не создастся вовсе.
- Каждый пункт хранит `sourceRequirementId` только для справки/аудита (не отдаётся в API) —
  после создания пункт полностью независим от изменений/удаления исходного
  `ProgramRequirement` (`ON DELETE SET NULL`).

**Честное ограничение этого этапа:** «авто-отметка по уже одобренным shared-документам»
из Roadmap реализована как точка расширения (`findApprovedSharedDocumentTypeIds`), но сейчас
возвращает пустой набор — модуля `Documents`/`ApplicationDocuments` ещё не существует
(появится в Фазе 4). Ничего не отмечается автоматически до тех пор.


## Applications API

Центральная сущность продукта — заявка связывает пользователя, университет, программу и цикл
поступления. Ни один роут не публичен (в отличие от каталогов) — это персональные данные.

- `POST /applications` — создание. `title` генерируется на backend автоматически
  (`"Университет — Программа — Цикл"`), не принимается от клиента. Уникальность по
  (`userId`, `universityId`, `programId`, `admissionCycleId`) — повторная попытка даёт 409.
- `GET /applications` — список: обычный пользователь видит только свои заявки; Mentor/Admin/
  SuperAdmin — глобальный доступ (решение по Mentor, раунд 2), с опциональным фильтром
  `userId`.
- `GET /applications/:id` — доступ только владельцу или роли с глобальным чтением, иначе 403.
- `PUT /applications/:id/status` — смена статуса. Финальные статусы (`Accepted`/`Rejected`/
  `Enrolled`) может выставить только Admin/SuperAdmin — это общее правило, а не только
  ограничение для Mentor (сам студент тоже не может проставить себе "Accepted").

**Отклонение от исходной схемы:** поле `currentStage` не хранится в БД — вычисляется на лету
из `applicationStatus` (`currentStageLabel` в ответе API), чтобы не плодить рассинхронизацию
между статусом и его текстовым описанием.

## Деплой

Roadmap, Этап 14.2 — последний этап всего плана. Конфиги (`frontend/vercel.json`,
`backend/railway.json`, `backend/Procfile`) рассчитаны на связку Vercel (frontend) / Railway
(backend) / Supabase (БД), как и было решено на этапе архитектурного планирования.

### 1. База данных (Supabase)

1. Создать проект на [supabase.com](https://supabase.com).
2. Settings → Database → Connection string → скопировать URI. Это будущий `DATABASE_URL`.

### 2. Backend (Railway)

1. Создать проект на [railway.app](https://railway.app), подключить GitHub-репозиторий, указать
   `backend/` как корневую директорию сервиса.
2. Settings → Variables — задать **все** переменные из чек-листа секретов (см. Этап 14.1 выше),
   реальными значениями (не из шаблона `.env.production` — тот коммитится пустым намеренно).
3. Railway автоматически подхватывает `railway.json`: сборка через Nixpacks (`npm run build`),
   запуск — `npm run prisma:migrate:deploy && npm run start:prod`. **Миграции применяются
   автоматически при каждом деплое** — вручную прогонять не нужно (`migrate deploy`, в отличие
   от `migrate dev`, идемпотентен и безопасен на каждый рестарт: применяет только новые
   миграции, ничего не спрашивает интерактивно).
4. **Важное ограничение MVP**: аватары пользователей хранятся на локальном диске backend-
   контейнера (`/uploads/avatars`), не в Google Drive — без **persistent volume** они не
   переживут следующий редеплой. Railway → Settings → Volumes → примонтировать на `/app/uploads`.
5. После первого успешного деплоя скопировать публичный URL backend (например
   `https://xxx.up.railway.app`).
6. Выполнить bootstrap SuperAdmin — миграции применяются автоматически, но seed нужно запустить
   вручную один раз через Railway CLI:
   ```bash
   railway run npm run prisma:seed
   ```

### 3. Frontend (Vercel)

1. Импортировать репозиторий на [vercel.com](https://vercel.com), указать `frontend/` как Root
   Directory. `vercel.json` подхватится автоматически (framework: vite, build/output уже заданы,
   `rewrites` обеспечивает работу client-side роутинга React Router при прямом переходе/обновлении
   страницы на любом пути).
2. Задать переменную окружения `VITE_API_BASE_URL` = URL backend из шага 2.5 + `/api/v1`
   (например `https://xxx.up.railway.app/api/v1`).
3. Деплой.

### 4. Вернуться в Railway — задать CORS

`CORS_ORIGIN` = URL frontend из Vercel (например `https://xxx.vercel.app`). Без этого браузер
будет блокировать все запросы с фронтенда на backend (Этап 0.2/12.1: CORS настроен на
конкретный origin, не wildcard).

### 5. Смоук-тест в production (вручную — Roadmap, Этап 14.2)

Регистрация недоступна (платформа закрытая) — проверяется вручную, не автоматизированным тестом:

- [ ] `GET /health` напрямую по URL backend отдаёт `200` без токена.
- [ ] Логин под bootstrap SuperAdmin проходит, cookie устанавливается.
- [ ] Создание университета/программы/цикла поступления через `/admin`.
- [ ] Создание второго пользователя (студента) через `/admin/users`, вход под ним.
- [ ] Создание заявки студентом.
- [ ] Загрузка документа — проверить, что файл реально появляется в Google Drive (не только
      успешный HTTP-ответ) и что AI-проверка (Этап 6.1) через некоторое время проставляет статус.
- [ ] AI-чат — реальный ответ от OpenAI, не ошибка авторизации/квоты/таймаута.
- [ ] Чек-лист обновляется после одобрения документа.

### Известные ограничения MVP при деплое

- **Аватары на локальном диске** — см. п. 2.4 выше, нужен persistent volume.
- **Нет CI/CD пайплайна** — папка `.github/` зарезервирована с самого Этапа 0.1, но не
  наполнена: деплой запускается автоматически по push в main через встроенную интеграцию
  Vercel/Railway с GitHub, без отдельного pipeline, прогоняющего unit/e2e-тесты (Фаза 13) перед
  деплоем. Тесты нужно запускать вручную перед мержем в main.
- **E2E-тесты требуют отдельной тестовой БД** (Этап 13.2) — не направлены на production/staging
  автоматически ни при каком деплое.

## Users API

- `GET /users/profile` — полный профиль (в отличие от `GET /auth/me`, который отдаёт только
  минимум для проверки сессии).
- `PUT /users/profile` — частичное обновление (firstName/lastName/phone/country/city/birthDate).
- `PUT /users/avatar` — загрузка изображения (`multipart/form-data`, поле `avatar`, до 2MB,
  JPEG/PNG/WEBP). **Ограничение MVP:** хранится на локальном диске backend-контейнера
  (`/uploads/avatars`), не в Google Drive. На Railway это требует persistent volume — без
  него аватары не переживут редеплой.
- `PUT /users/password` — смена пароля (проверка текущего, инвалидация всех сессий кроме
  текущей).

## Universities API

- `GET /universities`, `GET /universities/:id` — публично (пагинация `page`/`limit`, поиск
  `search` по имени, фильтр `country`). Всегда только `isActive: true`.
- `POST /universities`, `PUT /universities/:id`, `DELETE /universities/:id` — только
  Admin/SuperAdmin. `DELETE` — soft-delete (`isActive=false`): начиная с Фазы 3 на университет
  ссылаются заявки, а с Этапа 2.2 — программы.

**Отклонение от исходного ТЗ:** эндпоинты `/universities/my`, `/universities/select`,
`/universities/remove` не реализованы — они относились к сущности `UserUniversities`,
заменённой на `Application` (см. решения по Части 6). Эта логика — в модуле Applications
(Фаза 3), а не здесь.

## Programs API

- `GET /programs`, `GET /programs/:id` — публично (пагинация, фильтры
  `universityId`/`degreeLevel`, поиск `search`).
- `POST/PUT/DELETE /programs/:id` — только Admin/SuperAdmin. `DELETE` — soft-delete, как и у
  University (с Фазы 3 на программу ссылается `Application.programId`).
- `GET/POST/PUT/DELETE /programs/:id/requirements[/:requirementId]` — вложенный ресурс
  (шаблон чек-листа). В отличие от самой программы, `DELETE` требования — **настоящее**
  удаление: при создании заявки требования копируются в `ChecklistItems` как независимый
  снепшот, поэтому у шаблона нет долгоживущих внешних ссылок.

Добавлена таблица `DocumentType` (Часть 2, п.11 ТЗ, предзаполнена через `prisma/seed.ts`) —
понадобилась раньше срока (Фаза 4) как FK-справочник для `ProgramRequirement.documentTypeId`.

## AdmissionCycles API

- `GET /admission-cycles`, `GET /admission-cycles/:id` — публично (пагинация, фильтры
  `year`/`season`). Глобальный справочник, не привязан к университету.
- `POST/PUT/DELETE /admission-cycles/:id` — только Admin/SuperAdmin. `DELETE` — soft-delete,
  как у University/Program (с Фазы 3 на цикл ссылается `Application.admissionCycleId`).
- Валидация: `endDate` обязан быть позже `startDate`; уникальность по паре (`season`, `year`).

## Личный кабинет (Frontend)

После входа пользователь попадает в layout с `Sidebar` (навигация) и `Navbar` (аватар, имя,
выход). Раздел `/settings` — три независимые формы (личные данные, аватар, смена пароля) на

заявок/дедлайнов/уведомлений появится по мере реализации соответствующих модулей (Фазы 3, 7, 8).

## Universities/Programs (Frontend)

`/universities` — список с поиском (debounce 300ms) и пагинацией. `/universities/:id` —
карточка вуза (рейтинг, стоимость обучения, сайт) со списком его программ. Логотип вуза —
внешний URL (Admin вставляет ссылку напрямую), в отличие от аватара пользователя не требует
префикса `getApiOrigin()`.

## Правила разработки

Полное техническое задание — во внутренней документации проекта (архитектура, стек, БД,
дизайн-система, правила разработки). Проект реализуется строго поэтапно согласно
`ZhanaGasyr_Roadmap.md`: каждый этап должен быть завершён, проверен и рабочим, прежде чем
начинается следующий.

Ключевые принципы (обязательны на каждом этапе):
- TypeScript strict mode, `any` запрещён;
- SOLID, DRY, KISS, Clean Architecture, Feature-Based Architecture;
- вся бизнес-логика — только в Service, контроллеры «тонкие»;
- Frontend никогда не обращается к БД, Google Drive или OpenAI напрямую — только через
  Backend REST API;
- никаких секретов в репозитории — только через переменные окружения.
